# 버전과 완료 기준

상태: 초안
목적: 작은 작업들이 어떤 기준으로 플레이 가능 버전 또는 릴리즈가 되는지 정의한다.

## 핵심 기준

게임은 기능 목록이 아니라 플레이 경험이다.

따라서 버전은 "몇 개의 기능이 구현됐는가"가 아니라 "어떤 플레이어 경험이 검증됐는가"로 판단한다.

## 계층

```text
게임 컨셉
  ↓
게임 설계 원칙
  ↓
시스템 설계
  ↓
시스템 구성요소
  ↓
작업 항목 / 기능
  ↓
플레이 가능한 세로 단면
  ↓
마일스톤
  ↓
버전
  ↓
릴리즈
```

## 단계 정의

| 단계 | 의미 | 완료 기준 |
|---|---|---|
| 게임 컨셉 | 만들 게임의 방향 | 한 줄 소개, 목표 플레이어, 플레이어 판타지 승인 |
| 게임 설계 원칙 | 게임이 지켜야 할 핵심 가치 | 원칙별 검증 기준 존재 |
| 시스템 설계 | 규칙과 구성요소 | 시스템 문서와 의존성 정의 |
| 작업 항목 | 실제 제작 작업 | 구현/검증 보고서 통과 |
| 세로 단면 | 핵심 경험을 작게 증명하는 플레이 가능 구조 | 시작-진행-목표 달성이 가능하고 세로 단면 검증 통과 |
| 마일스톤 | 제작 관리 단위 | 범위, 리스크, 알려진 문제, 다음 범위 정리 |
| 버전 | 이름 붙일 수 있는 플레이 가능 프로젝트 상태 | 버전의 약속과 검증 기준 충족 |
| 릴리즈 | 배포 단위 | 빌드, 패키징, QA, 릴리즈 노트 기준 충족 |

## 버전 예시

| 버전 | 의미 |
|---|---|
| `v0.1_concept_proof` | 컨셉, 게임 설계 원칙, 핵심 반복 구조가 문서와 간단한 프로토타입 기준으로 정리됨 |
| `v0.2_first_playable` | 시작-플레이-목표 달성까지 가능한 첫 플레이 반복 구조 |
| `v0.3_systems_prototype` | 주요 시스템이 최소 형태로 연결됨 |
| `v0.5_vertical_slice` | 목표 경험을 대표하는 짧은 완성형 구간 |
| `v0.8_content_alpha` | 콘텐츠 반복 제작 파이프라인이 동작함 |
| `v1.0_release_candidate` | 릴리즈 기준 충족 |

## 버전 완료 규칙

AI는 작업 항목을 많이 만들었다고 버전이 완성됐다고 말하면 안 된다.

버전 완료 조건:

1. 버전 문서가 존재한다.
2. 플레이어에게 약속하는 경험이 명확하다.
3. 포함 게임 설계 원칙, 시스템, 세로 단면이 명시되어 있다.
4. 아트, 개발, UX, QA 산출물이 제작 추적표로 연결되어 있다.
5. 언리얼 프로젝트 검증 또는 플레이테스트 근거가 있다.
6. 알려진 문제가 버전 문서에 기록되어 있다.
7. 다음 버전으로 넘길 범위가 분리되어 있다.

## 첫 목표

첫 대상은 `v0.2_first_playable`이다.

의미:

> 선택한 게임 컨셉의 핵심 반복 구조를 아주 짧게라도 플레이 가능한 형태로 만든다.

이 버전은 최종 아트나 많은 콘텐츠 양을 요구하지 않는다. 하지만 플레이어에게 약속하는 경험은 실제 플레이테스트로 확인되어야 한다.
